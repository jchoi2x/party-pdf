# Planning Guide

A collaborative PDF document editing platform that enables users to upload PDF files and access them through shareable deep links, with real-time viewing powered by Apryse WebViewer.

**Experience Qualities**:
1. **Seamless** - Users should transition smoothly from upload to collaboration without friction
2. **Professional** - The interface should feel polished and trustworthy for document work
3. **Collaborative** - Identity and presence should be clear, making multi-user interaction intuitive

**Complexity Level**: Light Application (multiple features with basic state)
  - This is a focused application with PDF upload, routing, identity management, and document viewing features, but without complex real-time collaboration logic or advanced state management needs.

## Essential Features

**PDF Upload & Submission**
- Functionality: User selects a PDF file via file input and submits to generate a unique collaboration session with persistent storage
- Purpose: Initiates the document collaboration workflow with file blob stored in IndexedDB
- Trigger: User clicks file input, selects PDF, then clicks submit button
- Progression: Landing page → File selection dialog → File selected state → Submit click → Generate ULID session ID → Store file blob in IndexedDB with session ID → Navigate to `/document/:id` with generated session ID
- Success criteria: PDF file blob is stored in IndexedDB, unique session ID generated via ULID, user navigated to `/document/:id` route, file persists through page reloads
- Technical Implementation: Uses ULID library for unique ID generation, IndexedDB for blob storage with custom database wrapper (`pdf-collaboration-db` database, `documents` object store with `id` keyPath)

**Unique Document Sessions with Persistence**
- Functionality: Each uploaded PDF gets a unique identifier and dedicated URL, with blob data stored in IndexedDB for persistence across page reloads
- Purpose: Enables sharing, persistence, and reloading of document sessions without re-upload
- Trigger: Form submission with valid PDF file
- Progression: Generate ULID → Store PDF blob in IndexedDB with metadata (id, name, blob, createdAt) → Navigate to `/document/:id` → On page reload, retrieve blob from IndexedDB using session ID from URL
- Success criteria: URL contains unique session ID that persists in IndexedDB, can be shared and accessed directly, document reloads from IndexedDB on page refresh without requiring file re-upload
- Technical Implementation: IndexedDB stores `StoredDocument` objects with structure `{ id: string, name: string, blob: Blob, createdAt: number }`, URL parameter carries session ID for retrieval

**User Identity Management**
- Functionality: Prompts user for name on first visit, stores in localStorage (not IndexedDB), allows inline editing
- Purpose: Identifies collaborators in the document view and sets user identity for Apryse annotations
- Trigger: User enters collaboration view without a stored name in localStorage
- Progression: Check localStorage for 'userName' → Show required name prompt dialog if empty (cannot be dismissed) → User enters name → Store in localStorage → Display name in header with avatar and badge → Enable inline editing via pencil icon
- Success criteria: Name persists across sessions via localStorage, prevents repeated prompts, can be edited via UI with immediate feedback, name is passed to Apryse WebViewer's `setCurrentUser()` for annotation attribution
- Technical Implementation: Uses localStorage with key 'userName', Dialog component prevents dismissal (blocks outside clicks and escape key), inline edit mode with input field and save/cancel actions

**PDF Document Viewer with Apryse WebViewer**
- Functionality: Renders PDF using Apryse WebViewer (v11.11.0) with collaboration features enabled
- Purpose: Provides the core document viewing and annotation experience powered by Apryse
- Trigger: User navigates to document route with valid document session ID
- Progression: Load document blob from IndexedDB using session ID → Create blob URL → Initialize Apryse WebViewer with CDN path → Load PDF from blob URL → Set current user for annotations → Revoke blob URL after document loads
- Success criteria: PDF displays correctly in Apryse viewer with full annotation capabilities, user identity is set for collaboration, blob URL is properly cleaned up, license key authentication succeeds
- Technical Implementation: 
  - Uses Apryse WebViewer library via npm package `@pdftron/webviewer@11.11.0`
  - CDN path: `https://cdn.jsdelivr.net/npm/@pdftron/webviewer@11.11.0/public`
  - License key: `demo:1773251044163:637ef9590300000000e0776822862dfcea1362e5ec2c24eef968e7609f`
  - Creates blob URL with `URL.createObjectURL()` and revokes with `URL.revokeObjectURL()` after document loads
  - Uses `annotationManager.setCurrentUser(userName)` to attribute annotations
  - Initializes in React ref with proper lifecycle management in `useEffect`

## Edge Case Handling

- **Invalid File Type**: Show error toast if user selects non-PDF file (validates `file.type !== 'application/pdf'`)
- **Missing Document ID**: Redirect to home if document ID doesn't exist in IndexedDB, show error toast
- **Name Dialog Dismissal**: Prevent closing without entering a name (required field) - blocks `onPointerDownOutside` and `onEscapeKeyDown` events
- **Apryse Load Failure**: Show error toast if WebViewer fails to initialize, log error to console
- **Empty File Selection**: Disable submit button until file is selected
- **Empty Name Submission**: Show error toast if user tries to save empty name in dialog or edit mode
- **Page Reload**: Document blob is retrieved from IndexedDB using session ID from URL parameters, allowing seamless continuation
- **Blob URL Cleanup**: Blob URLs are revoked after WebViewer document loads to prevent memory leaks

## Design Direction

The design should evoke trust, clarity, and focus—creating an environment conducive to document work. Professional without being sterile, with subtle warmth that makes the tool approachable.

## Color Selection

A sophisticated palette centered on deep blues and warm accents that balance professionalism with approachability.

- **Primary Color**: Deep Navy Blue (oklch(0.35 0.08 250)) - Conveys trust, professionalism, and focus
- **Secondary Colors**: Soft Slate (oklch(0.55 0.02 250)) for supporting UI elements, Light Cream (oklch(0.97 0.01 85)) for subtle backgrounds
- **Accent Color**: Warm Amber (oklch(0.70 0.15 65)) - Draws attention to CTAs and active states
- **Foreground/Background Pairings**: 
  - Primary Navy (oklch(0.35 0.08 250)): White text (oklch(0.98 0 0)) - Ratio 8.9:1 ✓
  - Accent Amber (oklch(0.70 0.15 65)): Dark Navy text (oklch(0.25 0.08 250)) - Ratio 8.2:1 ✓
  - Background Cream (oklch(0.97 0.01 85)): Dark text (oklch(0.25 0.08 250)) - Ratio 13.1:1 ✓

## Font Selection

Typography should communicate precision and clarity while maintaining readability for extended document work sessions.

- **Primary Font**: Space Grotesk for headings and UI elements - brings technical precision with geometric character
- **Secondary Font**: Inter for body text and labels - ensures excellent readability at all sizes

- **Typographic Hierarchy**: 
  - H1 (Page Title): Space Grotesk Bold/32px/tight letter spacing
  - H2 (Section Headers): Space Grotesk Semibold/24px/normal spacing
  - Body (Form Labels, UI Text): Inter Regular/16px/relaxed line height
  - Small (Hints, Metadata): Inter Regular/14px/comfortable tracking

## Animations

Animations should guide attention and provide feedback without delaying workflow. Focus on micro-interactions that confirm actions and smooth state transitions.

- Upload button: Subtle scale on hover (1.02x) with 150ms ease
- File selection: Gentle fade-in of file name display
- Navigation transitions: Smooth 300ms page fade for route changes
- Name dialog: Modal slides in from top with backdrop blur effect
- Submit action: Loading spinner with pulsing animation during processing

## Component Selection

- **Components**:
  - `Card` for upload form container with subtle shadow
  - `Button` with primary variant for submit, ghost variant for name editing and home navigation
  - `Input` (file type) for PDF selection with custom styling to match theme
  - `Dialog` for name prompt with backdrop blur and dismissal prevention
  - `Input` (text type) for name entry with focus ring in accent color
  - `Label` for form field descriptions
  - `Avatar` with `AvatarFallback` showing user initials for identity display
  - `Badge` (secondary variant) to display current user name
  - Custom IndexedDB wrapper for blob persistence (`src/lib/indexeddb.ts`)
  - Apryse WebViewer component integration via npm package (@pdftron/webviewer v11.11.0)
  - React Router for navigation between upload and document views

- **Customizations**:
  - Custom file input styling with phosphor icons (`FilePdf`, `Upload`) and file size display
  - Apryse WebViewer wrapper with proper React lifecycle management (useRef + useEffect)
  - Custom header component showing document title, user avatar, name badge, and inline name editing
  - IndexedDB utilities: `saveDocument()`, `getDocument()`, `deleteDocument()`, `getAllDocuments()`
  - Name dialog with required field validation and blocked dismissal

- **States**:
  - Button: Disabled state when no file selected (reduced opacity, no hover effect), loading state with spinner during upload
  - File input: Empty vs. file-selected states with file info card (name, size, PDF icon)
  - Name display: View mode (avatar + badge + edit button) vs. edit mode (inline input + save/cancel buttons)
  - Dialog: Required field validation with error toast, cannot be dismissed until name is entered
  - Document page: Loading state with centered spinner while fetching from IndexedDB
  - WebViewer: Blob URL lifecycle management (create → load → revoke)

- **Icon Selection** (Phosphor Icons):
  - `Upload` for file upload action button
  - `FilePdf` (filled weight) for PDF file type indication in file preview
  - `PencilSimple` (bold weight) for edit name action
  - `House` (bold weight) for home/back navigation button
  - Avatar displays user initials (first 2 characters of name parts)

- **Spacing**:
  - Container padding: `p-4 sm:p-8` (16px mobile, 32px desktop)
  - Form field gaps: `space-y-6` (24px)
  - Header padding: `px-4 sm:px-6 py-4` (responsive horizontal, 16px vertical)
  - Card shadow: `shadow-lg border-2` for emphasis
  - Avatar size: `h-9 w-9` (36px)
  - Button min height: `h-12` for primary actions (48px)

- **Mobile**:
  - Single column layout on mobile (<768px)
  - Full-width upload card with reduced padding (`p-4`)
  - Stacked header elements (document title and user info wrap naturally)
  - Touch-optimized button sizes (min 44px height, maintained with `h-9` and `h-12` classes)

  - User name badge hidden on mobile (`hidden sm:flex`) to save space
